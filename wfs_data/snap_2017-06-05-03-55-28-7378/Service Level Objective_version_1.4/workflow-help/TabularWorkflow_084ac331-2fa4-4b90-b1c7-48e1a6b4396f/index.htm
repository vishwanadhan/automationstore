Introduction to Service Level Objective (SLO)

 

Service level objectives (SLOs) provide a quantitative means of describing and comparing the capabilities of systems and services. SLOs provide the physical basis for metering and translating storage services into other tangible metrics, such as Watts, area, and dollars. This enables application of a business model to the delivery of storage services, such as a Service Level Agreement.  The SLO framework also provides a means for providers of storage service to drive optimization of utilization, performance modelling and trending analysis, and asset planning. 

 

SLOs describe a variety of service objectives for storage items, including LUNs, files, objects, and file systems. The important SLOs that are part of this pack of Service Catalog include performance, space, and data protection. There are several other attributes of storage items that can be described by SLOs. These attributes include security, durability, and availability.

 

What Service Level Class (SLC) is

 

An SLC as defined by an SLC descriptor is a set of attribute-value pairs for a defined set of SLO attributes. While the possible space of all SLCs is the infinite cross product of all realizable combinations of SLO attributes, in practice only a small number of service level classes need to be defined.

As a storage architect, you should either define or select an existing SLC to map the desired functionality to manage your NetApp systems.

 

SLC categories in the Service Catalog solution


As part of this pack’s Service Catalog solution, each SLC is either a combination of Storage service level or Protection service level. The Storage service level includes performance, space efficiencies, and availability SLO dimensions. The performance implementation of this service level is through the functionality called, Adaptive QoS Policy Manager. The Protection service level includes data protection and retention policies.

Adaptive QoS Policy Manager

 

Two major trends in storage have emerged simultaneously.  At one extreme, modern flash-accelerated storage systems have become so fast that they are capable of delivering thousands of IO operations per second per terabyte of data stored.  On the other hand, the falling prices of spinning disks allow delivery of capacity with a very low cost per gigabyte per month.

 

Therefore, multiple service levels differentiated by unique IOPS/TB ratios allow IT organizations to offer storage service levels with different price and performance characteristics suited for different applications.  The price and performance characteristics can differ from one service level to another by an order of magnitude.

 

Adaptive QoS Policy Manager enables an IT organization to deliver predictable and agile services, aligning performance and cost to the needs of applications. It does this by enabling a range of service levels aligned to application IO requirements.

Why you should use the SLO pack

 

WFA SLO pack improves the productivity of storage providers by providing a set of predefined SLCs that can be modified based on your requirement.

 

The inherent flexibility of WFA makes it an ideal platform to deliver a Service Catalog solution. You can perform the following key tasks using the WFA SLO pack:

·         Create and manage SLCs

·         Map SLCs to storage service level or protection service level

·         Provision storage (volumes and LUNs) using SLCs

·         Provision storage (volumes and LUNs) using specific storage service level or protection service level

·         Performance monitoring and conformance using Adaptive QoS Policy Manager

 

The workflows in the WFA SLO pack can do the following:

 

·         Provides a solution (for the Storage Architect) to define the SLCs in terms of SLOs

Storage architects can define new service level classes, modify existing service level classes, or delete existing service level classes using the set of workflows in the SLO pack.

·         Enables you to map the service level objectives to concrete storage system properties using SLO workflows

·         Provides a set of workflows for provisioning of storage resources, such as LUNs and volumes with assigned SLCs to storage consumers

Workflows can provision storage resources by assigned SLOs and build intelligence to view mapping details and select the required resources and effectively provision the resources to meet the SLO definition.

·         Integrates the “Adaptive QoS Policy Manager” with WFA to enable SLO conformance and performance for the volumes provisioned.

This solution manages the QoS tool configuration and lifecycle. The Service Levels from the Adaptive QoS Policy Manager integrates into the Service Catalog within WFA.

Executing the workflows

 

You can execute the SLO workflows by using WFA.  You should download the workflow pack from storage Automation Store and import them from WFA for execution.

See the table for the sequence in which you must execute the workflows.

 

Before you begin:

·         You must have met the requirements specified.

·         You must have met the workflow-specific requirements mentioned in the table.

·         You must have acquired the required credentials for you to use WFA.

 

Steps:

1.    Log in to WFA by providing the necessary credentials.

2.    Click Portal > workflow_category > Workflow_name.

Based on the workflow you want to execute, select the category such as Service Level Management from the Portal.

3.    Click the workflow name.

The Execute Workflow: ‘Workflow_name’ dialog box is displayed.

4.    Provide the necessary input parameters required for execution of the selected workflow.

See the table that specifies the input parameters required for each of the workflow.

5.    Optional: Click Preview to validate your workflow before executing it.

6.    Click Execute.

You can also schedule the workflow for execution at a later date and time by selecting the Choose Date and Time for Execution check box.

Glossary

 

Service level objective (SLO) – SLOs are objectives that are achieved within a service level agreement (SLA).

Service level class (SLC) – SLCs are a set of service attribute-value pairs that are defined for a set of SLO attributes.

Quality of service (QoS) – QoS is a measure of service provided that ensures a certain performance level for a specific application or workload is always provided. For storage systems, this level is typically expressed as IOPS.

Adaptive QoS – The NetApp Adaptive QoS implements service levels with QoS.

Pack – A workflow pack is set of WFA workflows that are posted on the Storage Automation Store.

WFA – The NetApp storage management software that enables you to construct, customize, and use automated storage workflows for effective storage management.

Unified Manager – The NetApp storage management software that enables you to monitor your clustered Data ONTAP environment. Unified Manager is added as a data source in WFA.

Data source - A data source is a read-only data structure that serves as a connection to the data source object of specific data source type.

 

Edge – The protection relationship between the NetApp source and destination clusters is referred to as an edge in this document as well as the workflows.

 

An edge is defined by a set of the following parameters:

·         User-defined name of the edge

·         Edge type (disaster recovery, backup)

·         Edge source (source cluster or another edge)

·         Edge destination (storage service level class of the destination cluster)

·         Edge Recovery Point Objective (RPO) – specifies the last backup

·         Valid for backup edge only - Edge retention policy (specifies the number of Snapshot copies that can be retained in the destination cluster)

 

Protection Topology – Defines the protection setup in the SLO workflows, using the edges.
